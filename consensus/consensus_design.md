# 共识算法设计思想

把全球分成 M个战区，每个战区有 4个节点（分布在不同机房，互相备份），每次只有一个战区负责出块，打完 N 个区块后，接力棒传给下一个战区。
区域内 4 个节点的网络延迟非常低（同城/同区域）。BFT 类共识在 4 个节点之间收敛极快（毫秒级）
中立:  没有任何一个地区永远掌控链。欧美、亚洲、中东轮流坐庄，任何一个地区的监管机构都没有办法说"我们控制了这条链"。
N 个区块打完后，当前区域告诉下一个区域"你可以接手了"

其他区域的节点在"休息"时, 它们仍在实时同步区块、验证合法性，但不参与共识投票。相当于"监考官"，随时能发现活跃区域作弊。如果发现问题，可以拒绝接受这个区域的区块，触发异常处理。

活跃区域整体宕机怎么办？设一个超时机制：如果 T 秒内没有新区块，全球节点触发应急投票，直接跳到下一个区域。这是唯一需要全球协调的场景（但低频）。




## 共识核心
负责：在 Commonware 的 commonware-consensus 模块基础上，实现区域内 BFT 和轮次切换。

代码边界：

定义 Region / Node / Round 数据结构
用 Commonware 的 BFT primitives 配置动态验证人集合（每轮只激活当前区域的 4 个节点）
实现 Round Scheduler：区域顺序表维护、出 N 块后触发切换逻辑
生成 Handoff Block：包含状态根 + 下一区域 ID + 当前区域多签
超时容错：活跃区域无响应时，触发紧急跳轮逻辑
交付物：可单机跑通的 5 区域模拟环境，轮次切换稳定








## P2P 网络与跨区域同步
负责：在 Commonware 的 commonware-p2p 基础上，实现闲置节点的同步和监督。

代码边界：

配置跨区域 P2P 网络拓扑（各区域节点间的 peer 发现与连接）
实现区块广播：活跃区域出块后，实时推送到所有闲置区域节点
Watchdog 验证：闲置节点收块后执行状态验证，发现非法块广播 Fraud Proof
Handoff 多签收集：Dev 1 触发切换时，Dev 2 负责收集 ≥ 2/3 区域的确认签名
Catch-up 同步：节点重启后从最近 Handoff Block checkpoint 快速追赶
交付物：5 节点分布式测试网，闲置节点区块高度差 ≤ 3






## 治理与运维接口
负责：链上治理合约（或 runtime 模块）+ 外部接口，管理节点和区域的生命周期。

代码边界：

治理提案系统：AddRegion / RemoveRegion / AddNode / RemoveNode / UpdateParam(N, T) 提案类型，带公示期 + 投票期，在 Handoff Block 边界原子生效
配额约束检查：链上强制校验单司法管辖区节点占比不超过阈值
CLI / RPC 接口：节点注册、提案发起、投票、查询当前区域轮次状态
事件日志：治理变更历史可通过 RPC 查询
交付物：可通过命令行完成节点注册、提案投票、参数变更的完整流程

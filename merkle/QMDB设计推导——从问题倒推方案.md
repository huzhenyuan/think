# QMDB 设计推导——从问题倒推方案

> 不是"QMDB 有哪些设计"，而是"面对这个问题，我们应该怎么一步步想出这个设计"。
> 本文从区块链状态读写这个具体需求出发，逐步推导出 QMDB 的每一个核心机制，每个设计都有它必然存在的理由。

---

## 第一步：把问题说清楚

区块链状态数据库需要同时满足两件事，而这两件事天然矛盾：

**要快**：每个区块要处理几千笔交易，每笔交易读写几个账户。读写必须极快，否则出块速度跟不上。

**要可证明**：任何人拿着一个区块头（里面有一个状态根哈希），都能验证"某个账户在这个区块的余额是 X"——不需要信任节点，自己就能算出来。

这两件事的矛盾在于：

- 要快 → 用普通数据库（RocksDB、LevelDB 等）就够了，读写很快
- 要可证明 → 需要 Merkle 树，但 Merkle 树的每次写操作要更新从叶子到根的整条路径，以太坊大概 15 层，每层一次随机写，极慢

以太坊的选择是优先可证明，代价是读写都慢（每次读也要沿树走 15 层）。

**QMDB 的目标**：在不放弃可证明性的前提下，把读写速度提升到接近普通数据库的水平。

---

## 第二步：把"快"和"可证明"拆开

先问一个问题：**"可证明"是什么时候需要的？**

- 出块时（节点处理交易）：**不需要**证明，只需要读出正确的值
- 有人来查（轻客户端、跨链桥、审计）：**需要**证明

换句话说，"快速读写"和"生成证明"是两个独立的操作，完全可以由两套独立的机制负责。

**关键洞察**：

> 不需要在每次读写时都维护一棵完整的 Merkle 树。只需要在有人来查的时候，能够快速组装出一条证明路径。

这个洞察打开了设计空间——**读写路径和证明路径可以完全分离**，接下来分别解决。

---

## 第三步：解决"读写快"

### 3.1 把随机写变成顺序写

以太坊每次写一个账户，要更新 Merkle 树上的 15 个节点，每个节点都是一次磁盘随机写。随机写很慢，因为磁盘（包括 SSD）的随机写吞吐远低于顺序写。

**能不能把随机写变成顺序写？**

可以：**只追加，不修改**。

每次写操作不去更新旧记录，而是在文件末尾追加一条新记录（叫 Entry）。旧记录永远留在原位，永不修改。这样所有写操作都是顺序追加，SSD 的顺序写吞吐比随机写高一个数量级以上。

**副作用**：同一个账户会有很多条历史记录散落在文件里，不能直接定位"最新的那条在哪"。

### 3.2 解决读的问题：内存 B 树索引

如果只追加，要找某个账户的最新值，就得扫描整个文件——这更慢。

**解法：在内存里建索引**。

维护一张内存表（B 树）：`Key 的哈希前缀（9 字节）` → `最新 Entry 在文件里的字节偏移（6 字节）`。

查账户就是：查内存 B 树（不读磁盘）→ 拿到文件偏移 → 跳过去读一次磁盘。总共 **1 次磁盘读**。

**内存表有多大？**

每条记录 9 + 6 = **15 字节**（不存 Key 全文，不存 Value）。16 GB 内存能索引约 10 亿账户，够用。

**为什么只存 9 字节前缀而不是完整的 28 字节 Key？**

节省内存——每条记录省 19 字节，10 亿账户省出的内存量非常可观。代价是极小概率的前缀碰撞（两个不同 Key 的前 9 字节相同）。碰撞了就多读一次磁盘，对比完整 Key，正确性不受影响。

### 3.3 Shard：多核并行的基础

单个 B 树有读写锁竞争——多线程并发访问时退化为串行处理。

**解法：把 Key 空间切成 N 块（Shard），每块有独立的 B 树、独立的追加文件、独立的读写锁**。

具体做法：对应用层 Key 做一次哈希，得到均匀分布的 28 字节哈希值，取**前 4 位**（16 种可能值）决定分到哪个 Shard。

```
① 应用层 Key（如以太坊账户地址）
        ↓ Hash()
② 28 字节哈希值（均匀分布）
        ↓ 取前 4 位 → 决定分配到哪个 Shard（0 ~ 15 共 16 个）
        ↓ 取前 9 字节 → 在该 Shard 的 B 树里查找
③ 查内存 B 树 → 拿到磁盘偏移
        ↓ 一次 SSD 随机读
④ 读出 Entry → 核对 Entry 里存的完整 28 字节 Key 是否匹配
⑤ 匹配 → 返回 Entry.Value；不匹配 → B 树找下一条（哈希前缀碰撞，极罕见）
```

16 个 Shard = 16 个核可以完全并行处理不同账户的读写，互不阻塞。每个 Shard 也有自己独立的 Fresh Twig（后文详述）。

**为什么先 Hash 一次？**

应用层 Key（以太坊地址、合约槽位等）分布不均匀，直接切分会导致各 Shard 负载严重倾斜。哈希后分布趋向均匀，16 个 Shard 的 B 树大小和写入量都接近，负载自然均衡。

到这里，读写问题解决了：**写是顺序追加，读是 1 次磁盘读，16 个 Shard 并行互不干扰**。

## 第四步：解决"不存在证明"

能查到"账户存在时的值"还不够。还有个棘手的问题：怎么证明某个账户**不存在**？

如果直接说"我查了索引，没找到"——这不够，节点可以撒谎说"没找到"来隐藏账户。

**需要一个结构性的、可验证的不存在证明**。

如果所有活跃账户的 Key 按哈希值排好序，构成一条有序链表，那么：

> "alice 的下一个邻居直接是 carol"——意味着 alice 和 carol 之间，不可能有其他账户。

只要证明 alice 这条 Entry 是真实的（Merkle 路径验证），并且 alice.NextKey 确实 > bob.Key，就证明了 bob 不存在。

### 4.1 NextKey 字段

**每条 Entry 加一个 `NextKey` 字段**：哈希空间里紧挨在我后面的那个活跃账户的 Key。

所有活跃账户通过 NextKey 串成一条有序链表，两端由哨兵 MIN（0x00...00）和 MAX（0xFF...FF）封口：

```
MIN → alice(0x1A) → bob(0x3B) → carol(0x5C) → MAX
```

证明 dave（0x4D）不存在：找到 bob（bob.Key=0x3B < 0x4D < 0x5C=bob.NextKey），提供 bob 的 Merkle 路径证明，路径验证通过则 dave 确实不存在。

### 4.2 引入 NextKey 后的写操作精确代价

维护这条有序链表需要在写操作时额外更新邻居，代价如下：

| 操作 | 磁盘读 | 追加新 Entry |
|------|--------|-------------|
| Insert（新建账户） | 1 次（读前驱，获取前驱的 NextKey 和 OldId） | 2 条（新账户自身 + 前驱的新版本，更新前驱的 NextKey） |
| Update（修改账户） | 1 次（读当前版本） | 1 条（新版本，Value 更新，NextKey 不变） |
| Delete（删除账户） | 2 次（读被删账户获取其 NextKey + 读前驱） | 1 条（前驱的新版本，NextKey 跳过被删账户） |

这是必要的代价：**有序链表结构是不存在证明的基础**，没有捷径。

---

### Demo：从零开始建库（h=100 两笔 Insert）

假设三个账户的哈希排序：`Hash(alice)=0x1A < Hash(bob)=0x3B < Hash(carol)=0x5C`

初始状态有两个哨兵，代表空链表：

```
#0  MIN(0x00)  Value=–  NextKey=MAX  OldId=–   h=0
#⊤  MAX(0xFF)  （无后继）                        h=0
```

**新建 alice（余额=50）**：alice 的前驱是 MIN，读出 #0，追加 2 条：

```
#1  alice  Value=50   NextKey=MAX   OldId=–    ← alice 自身，后继暂时是 MAX
#2  MIN    Value=–    NextKey=alice OldId=#0   ← MIN 新版本：后继从 MAX 改成 alice
```

\#0 标记失效。内存索引：MIN→#2，alice→#1。

**新建 bob（余额=80）**：bob 的前驱是 alice，读出 #1，追加 2 条：

```
#3  bob    Value=80   NextKey=MAX   OldId=–    ← bob 自身，后继暂时是 MAX
#4  alice  Value=50   NextKey=bob   OldId=#1   ← alice 新版本：后继从 MAX 改成 bob
```

\#1 标记失效。内存索引：alice→#4，bob→#3。

**h=100 后完整日志**：

```
Id   账户    余额   NextKey   OldId   高度    有效？
──────────────────────────────────────────────────
#0   MIN      –     MAX       –       h=0     ❌
#1   alice   50     MAX       –       h=100   ❌
#2   MIN      –     alice     #0      h=100   ✅
#3   bob     80     MAX       –       h=100   ✅
#4   alice   50     bob       #1      h=100   ✅
```

当前有序链表：`MIN(#2) → alice(#4, 50) → bob(#3, 80) → MAX`

## 第五步：历史查询——OldId 与 OldNextKeyId

内存索引只指向最新版本，但我们需要能查到"账户在区块 H 时的值"，以及"账户在区块 H 时是否存在"。

### 5.1 历史版本的值：OldId 链

**每条 Entry 加 `OldId` 字段**，指向这个账户上一个版本的 Entry 编号。

每个账户的所有历史版本被串成一条时间链：

```
当前版本 #6（Version=350）
  └→ 上一版本 #5（Version=200）
       └→ 再上一版本 #4（Version=100）
```

给定 Entry 编号，直接算出磁盘位置（`编号 × Entry 大小`），一次跳转一次磁盘读，不需要扫描。

**Demo（续上）**：

区块 h=200，alice 余额改成 120，追加：

```
#5  alice  Value=120  NextKey=bob  OldId=#4   h=200
```

\#4 失效。内存索引：alice→#5。

区块 h=350，删除 bob（读 #3 得 bob.NextKey=MAX，读 #5 得 alice，追加前驱新版本）：

```
#6  alice  Value=120  NextKey=MAX  OldId=#5   h=350
```

\#3、#5 失效。内存索引：alice→#6，bob 从索引删除。

**查 alice 在 h=250 时的值**：

```
① 内存索引 → alice 当前是 #6，Version=350 > 250，需要往前
② 读 #6 → OldId=#5，读 #5，Version=200 ≤ 250，且下一版本(#6)的 350 > 250
   → #5 就是 h=250 的有效记录，Value=120 ✓
共 2 次磁盘读
```

### 5.2 历史时刻的不存在证明：OldNextKeyId

查"bob 在 h=400 是否存在"，需要找到 h=400 时刻 alice 的 NextKey。

但 alice.NextKey 随着账户的插入/删除而变化——bob 被删除后，alice.NextKey 从 bob 变成了 MAX。如果直接用当前 alice.NextKey=MAX 来证明 h=400 时 bob 不存在，看起来够用，但在更复杂的情况下（比如 bob 被删又在别处重建），需要可以严格重建任意历史时刻的链表结构。

**解法：每条 Entry 加 `OldNextKeyId` 字段**，指向"在记录我此刻的 NextKey 之前，前一个持有该 NextKey 变化信息的前驱 Entry"。

有了 OldId 和 OldNextKeyId 双链，任意历史高度 H 的完整有序链表都可以被重建：

- 沿 OldId 链找到 H 时刻的账户值
- 沿 OldNextKeyId 链找到 H 时刻的 NextKey 邻居关系
- 组合出"H 时刻的不存在证明路径"

### 5.3 完整的 Entry 字段定义

| 字段 | 大小 | 作用 |
|------|------|------|
| `Id` | 8 B | 全局递增编号，`Id ÷ 2048` = 所在 Twig，`Id % 2048` = Twig 内位置，无需额外索引即可定位磁盘位置 |
| `Key` | 28 B | 应用层 Key 的哈希值（均匀分布的 28 字节） |
| `Value` | 224 B | 状态值 |
| `NextKey` | 28 B | 有序链表中紧挨在后的活跃账户 Key |
| `OldId` | 8 B | 该 Key 的上一版本 Entry 编号（历史值回溯链） |
| `OldNextKeyId` | 8 B | 前一个记录了此 NextKey 变化的 Entry 编号（历史邻居关系回溯链） |
| `Version` | 8 B | 写入时的**区块高度 + 交易索引**（双精度，保证同一区块内多笔修改同一账户时版本有序） |

`Version` 精确到区块高度 + 交易索引，而非仅区块高度，是为了保证在同一区块内有多笔交易修改同账户时，每个中间状态都有唯一的版本号，历史查询可以精确到交易粒度。

## 第六步：证明存在——Twig 设计

现在能快速查到答案，但怎么证明这个答案是对的？

需要一条 Merkle 路径：从这条 Entry 的哈希，一路哈希到全局状态根。轻客户端对着区块头里的状态根验证这条路径，路径对了就说明数据真实。

**问题**：Merkle 路径上的中间节点从哪来？

如果每次查询都要从磁盘读路径上的所有中间节点——又回到了以太坊的老路，极慢。

### 6.1 把中间节点常驻内存

Merkle 树的中间节点是从叶子的哈希计算出来的，只要叶子哈希还在，随时可以重算。那为什么不把中间节点**常驻内存**？

- 每条 Entry 写入后，算一次哈希，存到对应内存位置，以后不改变
- 每当有新叶子写入，向上重算相关路径的中间节点，更新内存

构造 Merkle 路径所需的所有哈希都在内存里，**构造路径 = 纯内存操作，0 次磁盘读**。

### 6.2 内存太大？引入 Twig 分组

如果把整棵树的全部中间节点都放内存，内存占用 ≈ 叶子数量 × 32 字节。10 亿账户 × 32 字节 = 32 GB，仅 Merkle 中间节点就需要 32 GB——太多了。

**关键观察**：Entry 是追加写，一旦写入就不会修改。意味着一批 Entry 固定下来之后，它们组成的小 Merkle 树**根哈希永远不变**，内部节点也永远不变。

那就把 Entry 分组：每 **2048** 条 Entry 为一组，叫 **Twig**。

- Twig 写满后，2048 个叶子哈希永远固定 → Twig 根哈希永远固定
- 之后这个 Twig 的内部 Merkle 节点不再变化，**可以全部从内存释放**，只需保留 32 字节根哈希
- 多个 Twig 的根哈希再组成上层树，最终到全局状态根

**一个写满的 Twig**，原始数据约 512 KB（2048 × ~256B），内存里只需保留 32 字节根哈希——**压缩了 99.9%**。

**为什么是 2048？**

2048 = 2¹¹，Twig 内部 Merkle 树 11 层。这个数字在两个压力之间取平衡：

- 太小（如 64）：Twig 数量多，上层树节点多，内存占用大；Fresh Twig 刷盘频繁
- 太大（如 65536）：Fresh Twig 在内存里暂驻时间长，占用大；单 Twig 顺序读代价高

论文实测 2048 在实际工作负载下内存效率最优。

### 6.3 两层树结构

整棵 Merkle 树分成两层：

```
全局状态根
    │
    ├── 上层树（常驻内存）
    │       每个叶子 = 某个 Twig 的根哈希
    │       节点数 = Twig 数量级，约数百万个 × 32 B
    │
    └── 下层：每个 Twig 的小树（2048 个叶子）
                Fresh Twig：Entry + 内部节点全在内存
                Full Twig：只留 32 B 根哈希，内部节点已释放
```

### 6.4 区块结束时状态根怎么算

每个 Shard 始终有且只有一个 **Fresh Twig**（当前正在写入的 Twig），全局共 16 个（每 Shard 1 个）。

Fresh Twig 有固定 2048 个叶子槽位：已写入的槽用 Entry 哈希填充，**未写入的槽用约定的空哈希（null hash）填充**。

每次新 Entry 写入 Fresh Twig，其内部 Merkle 树立刻重算（全内存操作），新的根哈希沿上层树逐级传播：Fresh Twig 根 → Shard 根 → 全局状态根，整个链路均在内存完成。

因此，**区块结束那一刻，全局状态根已经是最新值，不需要任何额外的"结算"步骤**，直接提交至区块头。区块边界与 Twig 边界不需要对齐。

用前面 h=100 的数据（#0～#4 共 5 条 Entry），以 8 叶子 Twig 简化演示（实际是 2048 叶子）：

```
全在内存的 Twig 内部哈希树（8 叶子）：

         twig_root
        /          \

  H(0,1) H(2,3) H(4,5) H(6,7)

  + 上层树的路径（同样全在内存）
  → 最终到全局状态根

轻客户端验证：
  Hash(Entry#4) → 逐层哈希 → 对比区块头里的状态根 ✓
```

## 第七步：Twig 生命周期与内存管理

随着写入增加，越来越多的 Twig 积累，需要一套分级管理机制让内存占用保持可控。

### 7.1 四种状态

| 状态 | Entry 数据 | Merkle 节点 | 内存占用 |
|------|-----------|------------|---------|
| **Fresh** | DRAM | DRAM（完整） | ~640 KB（Entry 原始内容 + 内部 Merkle 节点） |
| **Full** | SSD | 不在内存（根哈希固定，无需保留） | **288 字节** |
| **Inactive** | SSD | 不在内存 | **288 字节** |
| **Pruned** | SSD | 不在内存 | **32 字节**（仅上层树中的该位置哈希） |

**Fresh → Full**：写满 2048 条 Entry 时触发：

1. 2048 条 Entry 原始内容异步刷到 SSD（~512 KB 顺序写）
2. 计算并保存 Twig 根哈希（32 B）
3. 初始化 ActiveBits 位图（256 B，全 1，表示全部 2048 条有效）
4. 内部 Merkle 节点**全部从内存释放**

**288 字节的两个用途**：

- **32 B 根哈希**：参与上层树计算，是全局状态根的来源。无论该 Twig 内有多少 Entry 还活着，根哈希永远不变（Entry 追加写、只增不改的保证）。
- **256 B ActiveBits 位图**：每一位对应一条 Entry 是否还是当前有效状态。账户被更新时，旧 Entry 所在 Twig 的对应 bit 被置 0。**ActiveBits 不影响根哈希**（根哈希是对所有 Entry 计算的，不论死活），但用于不存在证明的历史重建和 Compaction 判断。

**Full → Inactive**：当整个 Twig 的 ActiveBits 全部变为 0，即 2048 条 Entry 均已被后续版本覆盖或被删除。此时 Twig 里没有任何当前有效状态，但 288 字节仍需保留，因为根哈希还在上层树里占一个叶子位置。

**Inactive → Pruned**：当上层树中某个子树的所有后代 Twig 都变为 Inactive，该子树可被递归替换为一个固定哈希值，这些 Twig 的 288 字节一并释放。Pruned Twig 只剩 32 字节残留在上层树节点中。

### 7.2 为 Full Twig 里的 Entry 出证明（历史证明场景）

Full Twig 的内部 Merkle 节点已从内存释放。若需为其中某条 Entry 构造 Merkle 路径，需要：

1. 从 SSD 顺序读出该 Twig 的全部 2048 条 Entry（约 512 KB，**1 次顺序 IO**）
2. 在内存重建内部 Merkle 树，提取路径

这是 **1 次 SSD 顺序读（512 KB）**，不是 2048 次随机读，代价完全可接受。

### 7.3 Compaction：主动推进生命周期

**问题**：如果某个账户余额从不变化，它所在 Twig 的 ActiveBit 永远是 1，整个 Twig 永远停在 Full 状态，无法变 Inactive。仅靠用户有机更新（发送交易），绝大多数 Twig 永远不会变 Inactive——内存里会积累越来越多的 288 字节 Full Twig。

**解法：Compaction Worker**

一个独立的后台进程，周期性地把旧 Full Twig 里还活着的 Entry 主动复制到当前 Fresh Twig，然后把旧 Entry 标记为失效：

```
旧 Full Twig（以 Entry #0～#2047 为例）：
  Entry #0：alice，余额 100，ActiveBit=1（还活着）
  Entry #1：bob，  余额 200，ActiveBit=0（已被更新过）
  ...
  Entry #2047：carol，余额 300，ActiveBit=1（还活着）

Compaction 操作：
  1. 顺序读出整个旧 Twig（1 次 512 KB IO）
  2. 在 Fresh Twig 末尾追加新 Entry（alice，余额 100，OldId→#0）
  3. 旧 Entry #0 的 ActiveBit 置 0
  4. 同样处理 carol → 追加新 Entry，旧 bit 置 0

完成后：
  旧 Twig 的 ActiveBits 全为 0 → 变为 Inactive → 最终 Pruned
  alice 和 carol 的最新 Entry 已在新的 Fresh Twig，OldId 链保持完整
```

**Compaction 必须是确定性的**

在共识系统中，所有节点必须从同一初始状态出发，独立执行相同的操作序列，最终得到完全一致的状态根。如果 Compaction 的触发时机不确定（依赖时间、随机数等），不同节点的 Twig 结构会分叉，导致状态根不一致，共识崩溃。

QMDB 的解法：**Compaction 的触发条件是基于 Entry 数量的精确阈值**（如"某 Shard 的 Full Twig 中有效 Entry 比例低于 50%"）。这个条件由链上可验证的 Entry 计数唯一决定，不依赖任何外部随机因素，所有节点在完全相同的时机触发 Compaction，结果完全一致。

**Compaction 的代价**：读一次 Full Twig（512 KB 顺序读） + 为每条活跃 Entry 追加一条新 Entry（Fresh Twig 内存操作）。代价很低，可在后台平滑进行。

## 设计总结：四个机制，各司其职

QMDB 的四个核心机制分别解决四个独立的问题：

| 需求 | 机制 | 关键数据结构 |
|------|------|-------------|
| 读当前状态只读一次磁盘 | 内存 B 树索引 + 16 Shard 并行 | `Key前缀(9B) → 磁盘偏移(6B)` |
| 写操作快（顺序写，不更新旧数据） | 追加写 + Compaction（确定性触发） | 追加文件 + ActiveBits 位图 |
| 证明存在（当前 + 任意历史高度） | Twig 内存哈希树 + OldId 链 | Fresh/Full/Inactive/Pruned 生命周期 |
| 证明不存在（当前 + 任意历史高度） | NextKey 有序链表 + OldNextKeyId 链 | Entry 双向历史指针 |

**每个机制与其他机制完全正交**：

- 读写路径（B 树索引）不需要感知 Merkle 树的存在
- Merkle 证明路径（Twig）不需要扫描磁盘，零 IO（Fresh）或一次顺序 IO（Full）
- 历史查询（OldId/OldNextKeyId）不需要快照或回放，沿指针跳
- 并行处理（Shard）在所有路径上均起作用，且各 Shard 之间无共享状态

这种正交性是 QMDB 最重要的设计原则：没有一个机制因为另一个机制而付出额外代价，每个机制都在自己的代价边界内以最优方式运作。

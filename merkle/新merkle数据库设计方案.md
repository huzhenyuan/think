# 新 Merkle 数据库设计方案

---

# 一、设计目标与核心取舍

## 为什么不用传统全局 Merkle Tree？

以太坊使用 Merkle Patricia Trie（MPT）来组织全局状态。这个结构能对「任意账户在任意高度的状态」给出可验证证明，但代价极高：

| 维度 | 以太坊 MPT | 本方案 |
|------|-----------|--------|
| 每次状态写 | 更新路径上 ~15 个节点，~15 次哈希 + DB写 | 直接写一条 KV，O(1) |
| 存储放大 | 每条状态值对应树上 ~15 层节点，存储量数倍膨胀 | 一条值就是一条记录 |
| 状态读 | 沿树走 ~15 步（RLP解码 + DB读） | KV 直查，O(1) |
| 每块额外 IO | 1000笔 tx → 约 15000 次额外 DB 写 | 仅本块变更数量的小常数倍 |

**本方案的核心判断**：读写性能是链上最高频的操作瓶颈，全局 Merkle Tree 带来的写放大和存储膨胀不可接受。选择用性能换取部分证明能力，是有意识的工程取舍。

## 设计目标优先级

| 优先级 | 目标 |
|-------|------|
| ★★★★★ | 状态读写极致性能，KV 直查不依赖树结构 |
| ★★★★☆ | 写放大最小化，存储结构简单 |
| ★★★★☆ | 每块处理代价与全局状态规模无关 |
| ★★★☆☆ | 对本块内的变更可提供可验证证明 |
| ★★☆☆☆ | 链外可验证性作为可选能力，不作为核心约束 |

# 二、现有设计

## 基本结构

**状态存储**：多版本 KV
```
(key, height) → value
```
读取某个 key 在高度 h 的值：直接查 `(key, h)`，如没有则取最近的历史版本。完全不走树，查询极快。

**每个区块写入**：变更集小树
```
本块的变更 Δ = [(k1,v1), (k2,v2), ...]
对这些 key 建一棵小二叉 Merkle Tree → 得到 change_root
```
`change_root` 写入区块头，代表「本块所有状态变更的指纹」。

**全局状态根**：滚动哈希
```
state_root_h = H(state_root_{h-1} || change_root_h)
```
每块只做一次哈希运算，不涉及全量状态树的更新。

## 现有设计能做什么

| 能力 | 说明 |
|------|------|
| ✅ 状态读取（极快） | KV 直查，O(1)，与全局状态规模无关 |
| ✅ 证明「key K 在区块 h 被修改，新值是 V」 | 用 change_root_h 的 Merkle 路径即可 |
| ✅ 区块头完整性（state_root 链式哈希） | 任意节点可以验证区块链没有被篡改 |
| ✅ 链内合约互相调用读取状态 | 直接走 KV，安全且高效 |

## 现有设计不能做什么

**核心缺陷只有一个**：

> 无法证明「key K 在高度 h 的值是 V」，当 K 在 h 这个区块**没有发生变更**时。

换句话说：只能证明「改了什么」，无法证明「现在是什么状态」。

# 三、改进方案

## 能改进的部分：辅助修改历史索引

在不改变任何存储和区块结构的前提下，节点可以额外维护一个辅助索引：

```
key → [h1, h2, h3, ...]   // 该 key 被修改过的所有区块高度，按时间排序
```

这个索引存在 RocksDB 里，写入代价极低（每次状态变更附带写一条索引）。

**有了这个索引，能做什么？**

证明「key K 在高度 h 的值是 V」：

1. 查索引，找到 K 在 h 之前最近一次修改的高度 h'，对应值是 V
2. 查索引，找到 K 在 h 之后下一次修改的高度 h''（若不存在则说明 K 之后未再变更）
3. 提供 `change_root_h'` 中 K=V 的 Merkle 路径（证明 h' 时确实改成了 V）
4. 声明：h' < h < h''，因此高度 h 时 K 的值是 V

**这个证明有效吗？**

对**全节点**：有效。全节点可以独立扫描 h' 到 h'' 之间的所有 change_root，确认 K 没有出现，从而验证上述声明。

对**轻客户端（只持有区块头）**：**无效**。轻客户端没有办法验证「h' 到 h'' 之间 K 没有出现」这件事，因为验证这句话需要检查中间每个区块的 change_root 内容，而轻客户端没有这些数据。

**改进小结**：辅助索引让全节点之间的状态证明变得可行，但无法解决轻客户端验证问题。

# 四、仍然不能解决的问题

## 根本原因

问题的根源在于：**change_root 只承诺了「变更集」，没有承诺「全量状态」**。

一个区块头里的 state_root，只是前一个 state_root 和 change_root 的哈希组合。一个不下载全量历史变更的轻客户端，拿到 state_root 之后，无法从中提取出任意 key 的当前值。

这和传统 Merkle Tree（如 MPT）的根本区别在于：
- **MPT 的 state_root**：是所有当前状态 key-value 的 Merkle 树根，对任意 key 可以提供 O(log n) 的证明路径，轻客户端验证路径后即可相信节点返回的值。
- **本方案的 state_root**：只是变更历史的滚动哈希，无法从中导出任意 key 的证明。

## 受影响的场景

### 1. 轻钱包 / SPV 验证

手机钱包只存区块头，不下载完整链数据。它请求节点返回「账户余额 = 100」，然后需要一个 Merkle Proof 来独立验证这个返回值是真实的。

- 传统链：节点提供一条 O(log n) 的路径，钱包本地验证，通过则可信展示余额。
- 本方案：如果该账户很久没有变更，节点无法提供任何可验证的证明，钱包只能无条件相信节点返回的数据。钱包节点可以谎报任意余额而用户无从察觉。

**影响**：依赖 SPV 的轻客户端（手机钱包、浏览器插件钱包等）无法安全使用。

### 2. 跨链状态证明桥

跨链场景中，目标链上的合约需要以无信任方式验证：「源链上某账户在高度 h 的余额是 X」。

需要这种能力的跨链桥：
- **状态证明桥**（直接读取源链状态 Merkle Proof）：**无法接入**。无法给出任意高度的状态证明。
- **事件证明桥**（Cosmos IBC 类，只证明发生过某个事件）：可以部分支持（change_root 能证明某次转账事件存在），但无法证明当前状态，导致应用层逻辑受限。
- **多签桥**（Wormhole、Ronin 等）：可以接入，验证者观察事件即可，不依赖 Merkle Proof，代价是信任假设强（Ronin 因此被盗 6 亿美元）。
- **ZK 桥**：底层仍依赖对状态的零知识证明，源头还是需要状态承诺结构，本方案同样无法支持。

**影响**：与外部生态的去信任跨链交互受到严重限制。

### 3. 链外可验证查询（预言机、审计等）

链外系统（预言机、合规审计、数据分析等）有时需要以无信任方式查询某个合约状态在特定高度的值，并能独立验证返回结果。

本方案对这类场景的支持同样依赖「能找到最近一次变更高度 + 全节点验证」，链外验证者如果没有完整历史数据，无法独立完成验证。

**影响**：无法支持去信任的链外状态查询，外部系统只能信任节点返回值。

## 能正常工作的场景（对比）

| 场景 | 结论 | 原因 |
|------|------|------|
| 链内合约调用 | ✅ | 走 KV 直查，无需 Merkle Proof |
| 全节点运行链内业务 | ✅ | 全节点有完整历史，辅助索引可满足 |
| 中心化交易所（跑全节点） | ✅ | 全节点直接读 KV，确认最终性即可 |
| PoS 质押协议（链内全节点） | ✅ | 协议层由全节点维护，不依赖证明 |
| 多签跨链桥 | ⚠️ 可用，但信任假设强 | 不需要 Merkle Proof，但牺牲去信任化 |
| 安全轻钱包（SPV） | ❌ | 账户长期未变更时无法给出证明 |
| 状态证明跨链桥 | ❌ | 无法对任意高度任意 key 给出状态证明 |
| 去信任链外状态验证 | ❌ | 轻量验证者无法独立核实节点返回值 |

# 五、Merkle Tree 的必要性

## 本质问题：轻客户端需要什么？

轻客户端只持有区块头（约 80~200 字节），不下载完整数据。它面对的核心问题是：

> 给定一个区块头里的状态根，我能不能对任意 key 的当前值进行独立验证？

**Merkle Tree 给出了这个答案**：可以。从 key-value 到根的路径（O(log n) 个哈希）就是证明，轻客户端验证路径后，即可确认节点返回的值没有被篡改。

**本方案的状态根给不出这个答案**：state_root 只是变更历史的哈希滚动结果，无法反向推导出任意 key 的值，也无法验证节点提供的数据是否属于这个根。

## 为什么不能用其他结构替代？

原则上，只要能对「全量状态集合」做「固定大小承诺 + 可验证成员证明」，就可以替代 Merkle Tree：

| 方案 | 证明大小 | 缺点 |
|------|---------|------|
| Merkle Tree（哈希树） | O(log n) | 写放大 O(log n)，存储放大，本方案的出发点就是回避这个代价 |
| Verkle Tree（多项式承诺） | O(1) | 需要 pairing 密码学假设，计算量更重 |
| RSA 累加器 | O(1) | 需要可信设置或强数论假设，实现复杂 |

**结论**：不存在既不承诺全量状态、又能支持轻客户端验证的结构。这不是工程问题，是信息论约束——轻客户端要在不下载全量数据的前提下验证成员关系，必须有一个对全量数据的固定大小承诺，Merkle Tree 是这个承诺最简单且无需额外密码学假设的实现。

## 最终结论

本方案做出了一个明确的取舍：

> 放弃「对任意高度任意 key 的轻客户端可验证性」，换取「读写性能极致、存储简单、区块处理轻量」。

这个取舍是否合理，取决于应用场景：

- 如果是一个**以全节点为主**的高性能链，主要服务链内业务，跨链通过多签或可信节点完成 → **本方案完全可行**。
- 如果需要支持**安全的手机钱包**、**无信任跨链桥**、**去中心化预言机** → **无论如何都需要一个对全量状态的结构性承诺**，Merkle Tree（或 Verkle Tree 等等价结构）不可回避。

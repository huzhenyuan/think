# QMDB 怎么工作的——用大白话讲

---

## 先把答案说清楚

QMDB 要解决两个问题，**两个问题由完全不同的机制解决**，Twig 只管第二个：

### 问题一：查当前状态只读一次磁盘

**解法：内存索引。Twig 跟这个没关系。**

内存里有一张表（B 树），记录每个账户的**最新 Entry 在磁盘文件的第几个字节**。  
查 alice：查表 → 拿到字节偏移 → 跳过去读一次磁盘 → 拿到 Value。完了。

以太坊 MPT 要从树根沿 16 叉分支一路往下走，每层都要读一次磁盘，大概 15 次。

**索引有多大？**

这张表每条记录只存两样东西：
- Key 的哈希的前 9 个字节（不存完整 Key，更不存 Value）
- 对应的磁盘偏移（6 字节）

合计每个账户约 **15 字节**。以太坊主网目前约 2.5 亿个账户：

```
2.5 亿 × 15 字节 ≈ 3.75 GB
```

16 GB 内存可以索引超过 **10 亿**个账户。论文的目标场景就是 16 GB 内存的普通服务器。

**9 字节前缀会冲突吗？**

会，但冲突只影响性能，不影响正确性。原因：

1. **正确性靠磁盘上的 Entry 兜底**：每条 Entry 里存的是完整的 28 字节 Key（地址哈希全部内容）。查询时读出 Entry 后，必须核对 Key 字段是否完全匹配，不匹配就说明冲突了，再往 B 树下一条查。

2. **冲突概率极低**：9 字节 = 72 位，有 2^72 ≈ 4700 亿亿个槽位。2.5 亿个账户塞进去，生日悖论下两个 key 撞在同一个前缀的概率约 0.07%（平均 1000 次查询里不到 1 次多读一次磁盘）。

3. **为什么不存完整 Key？**：ETH 地址 20 字节，但 QMDB 存的是 `Hash(地址)` 的哈希，完整是 32 字节。存 9 字节做索引键，剩下的交给磁盘上的 Entry 核对，这是用极少量额外 IO 换大量内存的经典取舍。

作为对比，以太坊节点光 MPT 的状态数据本身就有几百 GB，而且每次查询要在磁盘上走 15 层。

### 问题二：证明某账户在历史高度 H 的值

**解法：OldId 链（找值） + Twig（构造 Merkle 证明）。**

- **找值**：每条 Entry 有一个 `OldId` 字段，指向上一个版本的 Entry 编号。沿链往前跳，就能找到任意历史版本。每跳一步读一次磁盘。
- **证明这个值没被篡改**：需要一条 Merkle 路径从这条 Entry 一路哈希到全局状态根。轻客户端用这条路径来验证。
- **Twig 的作用**：这条 Merkle 路径上的所有中间哈希节点都缓存在内存里——所以构造证明路径**不需要读磁盘**。这就是 Twig 的唯一目的：让构造 Merkle 证明的代价几乎为零。

## 二、Entry 是什么，怎么存到磁盘的

每次写操作（新建/修改/删除账户）都追加一条新 Entry，旧 Entry 永不修改。

每条 Entry 的关键字段：

| 字段 | 作用 |
|------|------|
| `Id` | 全局递增编号（从 0 开始，每条 Entry +1） |
| `Key` | 账户地址的哈希值 |
| `Value` | 账户的值 |
| `NextKey` | 哈希排序中紧挨在我后面的活跃账户的 Key |
| `OldId` | 这个账户上一个版本的 Entry 编号 |
| `Version` | 写入时的区块高度 |

Entry 按顺序追加写入磁盘文件。**关键规律：**

```
已知一个 Entry 的 Id，就能算出它在文件里的位置：
  Twig 编号 = Id / 2048      （每 2048 条 Entry 是一个 Twig）
  Twig 内偏移 = Id % 2048 × 每条Entry大小

→ 不需要任何额外索引，给定 Id 就可以直接跳到对应磁盘位置读数据
```

这是新数据和老数据都能快速定位的原因——无论是 `#6`（最新版）还是 `#1`（三年前的旧版），给定编号就能算出磁盘位置，一次读取。

## 三、演示数据：从零开始建库

假设三个账户的哈希排序：

```
Hash(alice) = 0x1A  <  Hash(bob) = 0x3B  <  Hash(carol) = 0x5C
```

初始有两个哨兵，代表"空的有序链表"：

```
#0  MIN(0x00) → MAX(0xFF)
#⊤  MAX(0xFF) → (无)
```

---

### 区块 h=100：新建 alice（余额=50）和 bob（余额=80）

#### 新建 alice

alice 插入链表，前驱是 MIN。读出 MIN 的 Entry(#0)，追加两条：

```
#1  alice  Value=50  NextKey=MAX  OldId=#0   ← alice 本身，邻居暂时是 MAX
#2  MIN    Value=-   NextKey=alice OldId=#0  ← MIN 更新：下一个邻居从 MAX 改成 alice
```

#0 标记失效（被 #2 取代）。内存索引更新：alice→#1，MIN→#2。

#### 新建 bob

bob 插入 alice 和 MAX 之间。alice 是 bob 的前驱，读出 alice 的 Entry(#1)，追加两条：

```
#3  bob    Value=80  NextKey=MAX   OldId=#1  ← bob 本身，邻居是 MAX
#4  alice  Value=50  NextKey=bob   OldId=#1  ← alice 更新：下一个邻居从 MAX 改成 bob
```

#1 标记失效。内存索引：alice→#4，bob→#3。

#### h=100 后的完整日志

```
Id   账户    余额   下一个邻居   上一版本   高度    有效？
──────────────────────────────────────────────────────
#0   MIN      -     MAX         -         h=0     ❌
#1   alice   50     MAX         #0        h=100   ❌
#2   MIN      -     alice       #0        h=100   ✅
#3   bob     80     MAX         #1        h=100   ✅
#4   alice   50     bob         #1        h=100   ✅
```

当前有序链表：`MIN(#2) → alice(#4,50) → bob(#3,80) → MAX`

---

### 区块 h=200：alice 余额改成 120

读出 alice 当前版本 #4，追加一条：

```
#5  alice  Value=120  NextKey=bob  OldId=#4
```

#4 标记失效。内存索引：alice→#5。（**1 次磁盘读**）

---

### 区块 h=350：删除 bob

删 bob = 让 alice 的 NextKey 跳过 bob 直接指向 MAX。  
读出 bob(#3) 和 alice(#5)，追加一条：

```
#6  alice  Value=120  NextKey=MAX  OldId=#5
```

bob(#3) 和 alice(#5) 标记失效。内存索引：alice→#6，bob 从索引删除。（**2 次磁盘读**）

#### h=350 后的完整日志

```
Id   账户    余额   下一个邻居   上一版本   高度    有效？
──────────────────────────────────────────────────────
#0   MIN      -     MAX         -         h=0     ❌
#1   alice   50     MAX         #0        h=100   ❌
#2   MIN      -     alice       #0        h=100   ✅
#3   bob     80     MAX         #1        h=100   ❌  ← bob 被删
#4   alice   50     bob         #1        h=100   ❌
#5   alice  120     bob         #4        h=200   ❌
#6   alice  120     MAX         #5        h=350   ✅
```

当前链表：`MIN(#2) → alice(#6,120) → MAX`

## 四、怎么读数据：新数据 vs 老数据

### 读新数据（alice 现在是多少？）

```
① 查内存索引（不读磁盘）：alice → Id=#6
② 算磁盘位置：#6 / 2048 = Twig 0，偏移 = 6 × EntrySize
③ 跳过去读一次磁盘，拿到 Value=120

共 1 次磁盘读。
```

### 读老数据（alice 在 h=250 是多少？）

```
① 查内存索引：alice → Id=#6（最新版）
② 读 #6（1次磁盘读）：Value=120，Version=h=350，OldId=#5
   350 > 250，还需要继续往前

③ 从 OldId 知道上一版本是 #5
   算 #5 的磁盘位置：#5 / 2048 = Twig 0，偏移 = 5 × EntrySize
   读 #5（1次磁盘读）：Value=120，Version=h=200，OldId=#4
   200 ≤ 250，而下一个版本(#6)的高度 350 > 250
   → h=250 时 alice 的有效 Entry 是 #5，Value=120 ✓

共 2 次磁盘读（#6 和 #5）。
```

**关键点**：`OldId` 是一个数字（Entry 编号），给定这个数字就能算出磁盘位置，所以跳历史链的每一步都是一次精确的磁盘读，没有扫描，没有遍历。

## 五、Twig 和"构造 Merkle 证明"的关系

找到了 Entry #5，值是 120，但怎么证明这是链上的真实数据而不是节点捏造的？

轻客户端只有区块头里的**全局状态根**（一个哈希值）。要证明 Entry #5 是真实的，需要给出一条路径：

```
Hash(Entry #5 内容)
  → 和 #4 的哈希合并 → H(4,5)
  → 和 H(6,7) 合并  → H(4,5,6,7)
  → 和 H(0,1,2,3) 合并 → twig_root（这 8 条/2048 条 Entry 的小树根）
  → 通过上层树   → 全局状态根
```

如果这条路径对得上全局状态根，说明 #5 真实存在于链上。

**这条路径上的每个中间哈希（H(4,5)、H(6,7)、H(0~3)……）都缓存在内存里**，不在磁盘上。

这就是 Twig：**把 2048 条 Entry 组成一个小 Merkle 树，这棵树的所有中间节点都常驻内存**。Entry 原始内容刷到磁盘，但哈希节点不刷——所以构造证明路径是纯内存操作，0 次磁盘读。

以 8 个叶子的示意：

```
全部在内存：
         twig_root
        /          \
    H(0~3)        H(4~7)
    /    \         /    \
  H(0,1) H(2,3) H(4,5) H(6,7)
  /  \   /  \   /  \   /  \
 h0  h1 h2  h3 h4  h5 h6  h_null
        ↑失效的         ↑失效的
        用零哈希代替

（Entry 原始内容：已刷到磁盘，内存不保留）
```

想要 #5 的证明路径？直接从内存拿兄弟节点：`[h4, H(6,7), H(0~3)]`，全程不读磁盘。

**Twig 写满后内存占用**：每个 Twig（2048 条 Entry）只需保留：
- 32 字节：`twig_root`（子树根哈希）
- 256 字节：`ActiveBits`（每一位表示对应 Entry 是否有效，用于重算 twig_root）  
- 哈希树的中间节点（若干 KB，用于出证明路径，必要时也可临时重算）

相比 2048 × ~256 字节的原始内容，压缩了 99% 以上。

## 六、证明 bob 在 h=400 不存在

`NextKey` 字段的用途：有序链表里，如果某条 Entry 的 NextKey 直接跳过了某个范围，就能证明那个范围里不存在任何账户。

现在 alice 的 #6：NextKey = MAX。

把 #6 和它的证明路径发给轻客户端：

```
✅ 路径验证通过：#6 在 h=400 的全局根下真实存在
✅ alice.Key(0x1A) < bob.Key(0x3B) < MAX(0xFF) = alice.NextKey
   → alice 和 MAX 之间没有任何账户 → bob 不存在 ✓
```

## 七、三个机制的分工总结

```
问题一「查当前值只读一次磁盘」
  ↓ 由
  内存索引（B 树）
  Key → 最新 Entry 的磁盘位置 → 1 次磁盘读
  Twig 与此无关。

问题二「证明历史值」
  ↓ 分两步

  步骤 A：找历史值
    内存索引找到最新 Entry → 沿 OldId 链往前跳
    每跳一步 = 用 OldId 算磁盘位置 + 1 次磁盘读
    Twig 与此无关。

  步骤 B：构造证明路径（Merkle path）
    需要从 Entry 到全局根的所有中间哈希
    这些哈希全在 Twig 的内存缓存里
    → 0 次磁盘读
    这是 Twig 唯一参与的地方。
```

## 八、对比以太坊

| 操作 | 以太坊 MPT | QMDB |
|------|-----------|------|
| 查当前值 | ~15 次磁盘读（从树根走到叶子） | **1 次**（内存索引直接定位） |
| 写一条状态 | ~15 读 + ~15 写 | **1~2 读 + 追加写**（顺序写性能高十几倍） |
| 构造 Merkle 证明 | 需要从磁盘读路径上每个节点 | **0 次磁盘读**（Twig 的内存缓存） |
| 查历史状态 | 需快照或回放历史块 | **沿 OldId 链跳，每步 1 次磁盘读** |
